# PKI

Source: `app/pki/*`, `app/routers/pki.py`, `app/models.py`
(`CertAuthority`, `Certificate`, `CAType`, `CertStatus`).

## Hierarchy

Three CA types (`CAType`): `root`, `intermediate`, `issuing`. A CA links to its
`parent` via `parent_id`; roots have no parent. `build_hierarchy()` renders the
full tree (root → intermediate → issuing) with issued leaf certs nested under
their issuing CA — **public data only** (never any private-key bytes), also
exposed at `GET /pki/tree.json`.

Rules enforced in `app/pki/ca.py`:

- An intermediate/issuing CA **requires** a parent; a root **cannot** have one.
- **Issuing CAs are created with `pathlen:0`** (may only sign leaves) unless
  overridden.
- `sign_csr` **refuses to issue leaf certs directly from a root** — leaves come
  from intermediate/issuing CAs.
- You cannot create a child under a parent that has no usable private key
  (an imported cert-only CA) — it raises a clear `ValueError`.

## Keys & the KEK envelope

Private keys are generated by `app/pki/keys.py`:

- **EC** curves: `secp256r1`, `secp384r1`, `secp521r1`.
- **RSA**: any bit size (public exponent 65537).

Every private key is serialized to PKCS#8 DER and wrapped with
`app/security/crypto.py` — **AES-256-GCM** using the app KEK, stored as
`nonce(12) || ciphertext||tag` in `CertAuthority.key_enc`. It is decrypted
transiently in memory for a single signing operation and dropped.

**Invariant (verified by tests):** no route or function serializes a CA or
appliance *private* key out of the app. `chain_pem`, `ca_cert`, `cert_pem` and
the hierarchy JSON all return public material only.

Certificate signature hashing uses **SHA-384** throughout.

### Extensions

- **CA certs**: `BasicConstraints(ca=True, path_length=…)` (critical),
  `KeyUsage(key_cert_sign, crl_sign, digital_signature)` (critical),
  Subject Key Identifier, and (for non-roots) Authority Key Identifier.
- **Leaf certs** (`sign_csr`): `BasicConstraints(ca=False)` (critical),
  `KeyUsage(digital_signature, key_encipherment, key_agreement)` (critical),
  `ExtendedKeyUsage(clientAuth, serverAuth)`, SKI, AKI, and a
  `SubjectAlternativeName` built from the supplied DNS/IP SANs.

## Creating a CA

`POST /pki/ca` (admin). Fields: `name`, subject (`cn`, `org`, `country`),
`ca_type`, optional `parent_id`, `valid_days` (default 3650), `key_type`
(`ec`/`rsa`), `key_params` (curve name or RSA bits). The CA's cert is built and
self- or parent-signed; the private key is KEK-wrapped and stored.

## Importing a CA

`POST /pki/ca/import` (admin). Accepts:

- **Pasted PEM** — certificate (and optionally a private key).
- **File upload** — a PEM bundle or a **PKCS#12** `.p12`/`.pfx` (with optional
  password). An uploaded file takes precedence over pasted PEM.
  `app/pki/material.py` normalizes both to PEM: PKCS#12 is detected by extension
  or by not starting with an ASCII armor line.

Import logic (`import_ca`):

- The certificate must be a CA (`BasicConstraints CA=true`), else it's rejected.
- If a private key is supplied, it **must match** the certificate's public key.
- CA type is inferred (`root` if self-signed, `issuing` if `pathlen==0`, else
  `intermediate`) unless overridden by the form.
- The importer auto-links to an existing parent CA whose subject equals this
  cert's issuer.
- Cert-only imports (`key_enc == b""`) set `has_private_key == False` and cannot
  sign; imported serial counters start high (`0x1000000`) to avoid colliding with
  externally-issued serials.

## Signing a CSR

`POST /pki/sign`. Parses and validates the CSR self-signature, then issues an
end-entity certificate from the chosen **issuing** CA (`valid_days` default 825,
plus comma-separated `san_dns`). The signed cert and the submitted CSR are stored
in `certificates`. Download a leaf cert (`/pki/cert/{id}/pem`), or leaf + chain
(`?fmt=chain`), and view/download a CA chain (`/pki/ca/{id}/chain`).

### Appliance keypairs

`app/pki/csr.py` `generate_leaf_keypair_and_csr()` can build an appliance key +
CSR server-side. The **private key PEM is returned to the caller exactly once and
never persisted** — CSR-based issuance (device-generated key) is preferred.

## Deletion & locking (delete-protection)

Both CAs and certificates have a `locked` boolean toggled by admins
(`POST /pki/ca/{id}/lock`, `POST /pki/cert/{id}/lock`).

**Deleting a certificate** (`POST /pki/cert/{id}/delete`, admin): the operator
must re-type the exact `serial`; a locked cert is refused. Only the leaf record
is removed — CAs are never touched.

**Deleting a CA** (`POST /pki/ca/{id}/delete`, admin): the operator must re-type
the exact CA `name`. `delete_ca()`:

- Refuses a locked CA.
- Refuses a CA that still has sub-CAs or issued certs **unless `cascade`** is set,
  which removes the whole subtree (descendant CAs deepest-first, then their
  issued certs).
- Refuses a cascade if any **locked** sub-CA or **locked** certificate exists
  anywhere in the subtree.

Deleting a CA also deletes its (KEK-encrypted) private key.
